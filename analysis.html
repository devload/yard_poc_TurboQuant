CH.1 TurboQuant란 무엇인가

Google Research가 ICLR 2026에서 발표한 벡터 양자화(Vector Quantization) 알고리즘입니다. 원래 목적은 LLM의 Key-Value Cache를 3~4비트로 압축하여 GPU 메모리를 6~8배 절약하는 것입니다.

알고리즘 핵심 (2단계)

단계연산목적
1단계랜덤 직교 회전(QR 분해) + Lloyd-Max 최적 양자화벡터 좌표를 Beta 분포로 변환 후 MSE-최적 코드북으로 양자화
2단계QJL 1-bit 잔차 프로젝션양자화 잔차의 부호만 저장하여 내적(inner product) 편향 보정

TurboQuant가 잘하는 것

고차원 벡터(수백~수천 차원)의 내적을 보존하면서 극단적으로 압축합니다. 개별 값이 약간 틀려도, 벡터 간 유사도(내적/코사인)는 정확하게 유지됩니다. 이것이 LLM의 Attention Score를 보존하는 핵심 원리입니다.

TurboQuant가 못하는 것

개별 값의 정확한 복원은 보장하지 않습니다. 랜덤 회전 행렬이 값을 "섞기" 때문에, 입력 52.7이 복원 시 36.8이 될 수 있습니다. 차원이 낮을수록(5~20) 이 문제가 심각해집니다.

AI에서는 왜 스칼라 8-bit 대신 TurboQuant를 쓰는가?

스칼라 8-bit의 문제 (AI)

4096차원 벡터에서 각 원소에 ±0.2% 오차가 있으면, 내적 계산 시 4096개 오차가 누적됩니다.

내적 오차 ≈ 개별오차 × √4096 ≈ 64배 증폭

TurboQuant는 회전으로 오차를 균등 분배하고 QJL로 내적 편향을 보정하여 이 누적 문제를 해결합니다.

모니터링에서는?

모니터링 데이터는 내적이 아니라 개별 값을 읽습니다.

"CPU가 몇 %인가?" → 정확한 값 필요
"이 서버와 비슷한 패턴의 서버는?" → 내적/유사도 필요

용도에 따라 다른 방법이 필요합니다.
값 저장 → 스칼라 8-bit
패턴 비교 → TurboQuant

CH.2 WhaTap YARD 데이터 구조

YARD란?

WhaTap 모니터링 플랫폼의 핵심 데이터 수집/저장 서버입니다. 에이전트로부터 메트릭, 트레이스, 로그를 TCP(포트 6610)로 수신하여 yardbase(시계열 DB)에 저장하고, Front/API(포트 7710)를 통해 대시보드에 제공합니다.

판단 기준

TurboQuant에 맞는 데이터

  • 차원 50 이상
  • 동일하거나 비슷한 스케일
  • 개별 값보다 패턴/분포가 중요
  • 유사도 비교(내적)에 사용

TurboQuant에 안 맞는 데이터

  • 차원 20 이하
  • 스케일이 크게 다른 필드 혼합
  • 개별 값 정확도가 중요 (API 응답에 직접 표시)
  • 문자열 / 바이너리 / 인덱스
  • 이미 정수이거나 범위가 극히 좁은 값

CH.5 핵심 발견: 왜 저장 압축에는 안 맞는가

PoC Phase 2에서 압축률 8.57x를 달성했지만, 실제 API 응답값을 비교하면 사용 불가능한 수준의 오차가 발생합니다.

원인: 랜덤 회전 행렬이 서로 다른 스케일의 값을 섞습니다.
CPU(0~100)와 TPS(0~10000)를 같은 벡터에 넣으면, 회전 후 각 좌표에 두 메트릭의 정보가 혼합됩니다. 양자화 오차가 역회전 시 원래 스케일로 증폭되어 돌아옵니다. LLM의 임베딩은 모든 차원이 비슷한 스케일(-1~1)이라 이 문제가 없습니다.

CH.6 튜닝 시도와 한계

TurboQuant의 오차를 줄이기 위해 5가지 튜닝을 시도했습니다.

튜닝 방법상대오차압축률결과
현재: TQ 4-bit, dim=528.17%2.9x부족
비트 증가: TQ 6-bit10.72%2.5x여전히 부족
비트 증가: TQ 8-bit7.13%2.2x여전히 부족
시간 배치: dim=10039.50%7.4x더 나빠짐
델타 인코딩 + TQ2200%6.1x완전 파탄
회전 제거: codebook만 8-bit0.69%1.2x정확 (압축률 낮음)
스칼라 8-bit0.49%4.0x최적
결론: 회전 행렬이 오차의 근본 원인입니다. 회전을 제거하면 오차가 0.69%로 떨어지지만, 회전을 빼면 TurboQuant가 아니라 그냥 codebook 양자화입니다. 모니터링 데이터 저장에는 단순한 스칼라 양자화가 구조적으로 최적입니다.

120차원, 동일 스케일(count), 패턴 비교 목적 — TurboQuant가 설계된 정확한 시나리오입니다.

가능한 기능:
• "지금 히트맵과 비슷한 과거 시점 찾기" (장애 재현 분석)
• "히트맵 분포가 갑자기 바뀌면 알림" (응답시간 분포 이상 탐지)
• "A 서비스의 히트맵과 비슷한 패턴의 다른 서비스" (영향 범위 추적)
