YARD 데이터와 이상 탐지

이 PoC의 목적은 압축이 아니라 이상 탐지입니다. YARD에 쌓이는 모든 데이터를 이상 탐지 관점에서 분류합니다.

이상 탐지에 사용하는 데이터

TurboQuant는 숫자 벡터의 패턴 변화를 감지합니다. float든 Decimal이든 숫자이면 벡터로 만들 수 있고, 벡터가 되면 이상 탐지 대상입니다.

카테고리데이터 타입벡터 차원이상 탐지 용도현재 상태
app_counterFloat 39개39 서버 상태 패턴 변화 (CPU, TPS, GC, 응답시간 동시 비교) 적용 완료
app_host_resourceFloat 10개10 호스트 리소스 변화 (메모리, 디스크, 네트워크) 적용 완료
app_counter_projectFloat 10개 + Decimal10+ 프로젝트 전체 상태 변화 적용 완료
app_proc_counter {m1},{m5}Float + Decimal가변 프로세스별 리소스 이상 적용 완료
app_proc_counter (raw)Decimal 위주가변 프로세스 이상 탐지 — Decimal도 벡터에 포함 가능 다음 단계
app_actx_meterFloat 2개 + Decimal가변 액티브 트랜잭션 폭증/급감 감지 다음 단계
agent_status_summaryDecimal만가변 에이전트 상태 변화 (갑자기 죽거나 추가되는 패턴) 다음 단계
hitmapint[120]120 응답시간 분포 변화 — TQ 최적 대상 다음 단계 (1순위)
alert_countDecimal가변 알림 빈도 패턴 변화 검토 필요
agent_dic메타데이터- 에이전트 이름 사전 — 패턴 없음 대상 아님
핵심: 압축 기준 ≠ 이상 탐지 기준 압축은 "float가 3개 이상이면 효율적"이라 float 부족한 데이터를 skip했습니다.
이상 탐지는 "숫자가 있으면 패턴을 볼 수 있다"이므로 Decimal, int도 벡터 대상입니다.

현재 12개 파일에서 TQ 인코딩으로 이상 탐지가 동작 중이고,
나머지 7개도 Decimal/int를 벡터에 포함하면 이상 탐지 대상이 됩니다.
특히 hitmap(int[120])은 동일 스케일 120차원으로 TQ에 가장 적합합니다.

다음 단계: 이상 탐지 확장

현재: Float 필드만 벡터화
app_counter(39차원), app_host_resource(10차원)
↓
다음 1: Decimal/int도 벡터에 포함
app_proc_counter, agent_status, app_actx_meter → 전부 탐지 대상
↓
다음 2: hitmap(int[120]) 벡터 인덱스
응답시간 분포 변화 탐지 — TQ 최적 대상, 120차원 동일 스케일
↓
다음 3: 크로스 인덱스 연관 분석
counter + host_resource + hitmap을 시간으로 묶어 근본 원인 추적

현재 TQ 인코딩 레코드 구조

TQ 인코딩 레코드 (.dat2 내):
  [int3 길이][0x03][origLen 2B][floatCount 2B][metaLen 2B][meta bytes][TQ compressed bytes]
                ↑
          VERSION_TQ 헤더

TQ compressed bytes:
  TurboQuantizer(4-bit, dim=floatCount).compress(floats)
  = [4B norm] + [floatCount × 4 / 8 bytes packed indices]

예: floatCount=39 → 원본 156B(39×4B) → TQ 24B(4B norm + 20B packed)

→ 이 벡터가 .tqi에 쌓이고, 베이스라인 대비 코사인 유사도로 이상 탐지

정상 vs 이상 시나리오

정상 구간: 5에이전트, 2초 간격
유사도 ~1.0 (베이스라인과 동일 패턴)
↓ 에이전트 5→20 급증
이상 구간: 20에이전트, 1초 간격
유사도 0.67~0.80 (패턴 변화 감지)
감지 원리 TurboQuant는 벡터의 "방향(패턴)"을 보존합니다.
에이전트가 5→20으로 늘어나면 메트릭 벡터의 방향이 바뀌고,
코사인 유사도가 떨어져서 자동으로 이상으로 판정됩니다.

개별 메트릭 threshold("CPU > 90%")가 아니라
전체 메트릭 패턴의 변화를 한 숫자(유사도)로 감지하는 것이 핵심입니다.

이상 감지 → 원인 분석

이상을 감지하는 것만으로는 부족합니다. "왜 이상한가?"까지 자동으로 따라가야 합니다.

분석 흐름: 5단계 드릴다운

1. 이상 감지
TQI 데몬: 코사인 유사도 0.77 → "패턴이 바뀌었다"
↓ 같은 시간에 다른 인덱스도 이상한가?
2. 동시 이벤트 수집
±5분 윈도우로 다른 .dat2 카테고리 스캔
app_counter + app_host_resource + app_proc_counter
↓ 공통으로 벗어난 메트릭은?
3. 공통 메트릭 분석
여러 서버에서 동시에 sql_time +121%, resp_time +98%
→ "SQL이 느려지면서 응답시간이 올라갔다"
↓ 왜 그 메트릭이 올라갔나?
4. 서버 상세 확인
DB 커넥션 풀 96% 포화, 대기 12건
힙 메모리 88%, 디스크 I/O 45%
↓ 어떤 쿼리/URL이 원인인가?
★ 5. 근본 원인
SELECT * FROM orders WHERE... (200ms → 8500ms, 43배)
/api/order 150ms→4200ms, /api/payment 200ms→3800ms

→ 패턴: sql_time↑ + resp_time↑ = "DB 쿼리 지연 → 응답시간 증가"

Step 4: 원인 추정 규칙

공통 메트릭 패턴추정 원인

httpc_time ↑ + resp_time ↑외부 API 지연
heap_use ↑ + gc_count ↑ + gc_time ↑메모리 부족 / GC 빈발
cpu ↑ (sql 정상)CPU 과부하
다수 메트릭 동시 급등트래픽 폭증 또는 에이전트 추가

→ ★ /api/payment 180ms→12000ms (외부 결제 API 타임아웃)
시나리오 3: 메모리 릭 → [감지] TQI 유사도 점진적 하락 (0.95→0.80→0.65)
→ [메트릭] heap_use +58%, gc_count +72%, gc_time +118%
→ [서버] 힙 94%, GC에 CPU 38%
→ ★ 메모리 릭으로 모든 URL 느려짐
핵심 TQI 벡터 인덱스는 "뭔가 이상하다"를 빠르게 감지하는 역할이고,
YARD의 기존 데이터(카운터, 슬로우 쿼리, 프로필)가 "왜 이상한가"를 설명하는 역할입니다.

감지(TQI) + 분석(YARD 기존 데이터)를 시간으로 연결하면,
이상 발생부터 근본 원인까지 자동으로 추적할 수 있습니다.

자동 분석 코드 (RootCauseAnalyzer)

PoC에 포함된 RootCauseAnalyzer와 DeepDrillDown이 이 과정을 자동화합니다.

// 이상 이벤트 수집
List<AnomalyEvent> events = collectFromAllIndexes(timeWindow);

// 시간으로 묶기
List<IncidentGroup> groups = analyzer.correlate(events);

// 심층 분석
for (IncidentGroup group : groups) {
    DrillDownReport report = DeepDrillDown.analyze(
        group,
        serverDetailProvider,  // DB 풀, 힙, GC 상세
        queryStatsProvider     // 슬로우 쿼리, 영향 URL
    );

    // 원인 체인 출력
    for (CauseChainStep step : report.causeChain) {
        log("[%s] %s", step.level, step.description);
    }
    // → [증상] 히트맵 분포 이상
    // → [전파] 서버 3대 sql_time +121%
    // → [원인] DB 풀 96% 포화
    // → ★ [근본원인] SELECT * FROM orders... 43배 느려짐
}
