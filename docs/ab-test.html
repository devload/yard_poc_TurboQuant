.dat2 레코드 구조

원본 레코드:
  [int3 길이][_writeMap 바이트: bool + decimal + FloatValue(4B) + ...]

TQ 인코딩 레코드:
  [int3 길이][0x03][origLen 2B][floatCount 2B][metaLen 2B][meta bytes][TQ compressed bytes]
                ↑
          VERSION_TQ 헤더

TQ compressed bytes:
  TurboQuantizer(4-bit, dim=floatCount).compress(floats)
  = [4B norm] + [floatCount × 4 / 8 bytes packed indices]

예: floatCount=39 → 원본 156B(39×4B) → TQ 24B(4B norm + 20B packed) = 6.5x 압축
판단 기준 float가 3개 이상인 레코드만 TQ 인코딩. 그 이하는 오버헤드가 커서 원본 유지.
읽기 경로에서 첫 바이트가 0x03이면 TQ 디코딩, 아니면 원본 그대로 — 하위호환 100%.

동작 흐름 1. 5초마다 yardbase의 최신 .dat2 스캔
2. 새 레코드 발견 → TQ 디코딩 → float 벡터 추출
3. TurboQuantizer로 압축 → .tqi에 append
4. 처음 200개로 베이스라인 벡터 구축
5. 이후 벡터마다 베이스라인과 코사인 유사도 비교

정상 vs 이상 시나리오

정상 구간: 5에이전트, 2초 간격
유사도 ~1.0 (베이스라인과 동일 패턴)
↓ 에이전트 5→20 급증
이상 구간: 20에이전트, 1초 간격
유사도 0.67~0.80 (패턴 변화 감지)
감지 원리 TurboQuant는 벡터의 "방향(패턴)"을 보존합니다.
에이전트가 5→20으로 늘어나면 메트릭 벡터의 방향이 바뀌고,
코사인 유사도가 떨어져서 자동으로 이상으로 판정됩니다.

개별 메트릭 threshold("CPU > 90%")가 아니라
전체 메트릭 패턴의 변화를 한 숫자(유사도)로 감지하는 것이 핵심입니다.

이상 탐지 로그 분석 방법

로그 위치

# B 서버에서
cat /data/whatap-server/logs/tq-daemon.log

# 이상만 필터

grep "유사도=" /data/whatap-server/logs/tq-daemon.log | \

유사도 수준별 의미

유사도의미조치
0.95~1.00정상 — 베이스라인과 거의 동일없음
0.80~0.95주의 — 약간의 패턴 변화모니터링
0.60~0.80이상 — 뚜렷한 패턴 변화원인 조사
< 0.60심각 — 완전히 다른 패턴즉시 대응

"유사 과거 벡터" 활용

⚠ 이상 #1 [13:32:30] tqi#202 유사도=0.7735
  유사: tqi#152 (1.0000)
                  ↑
  "과거 벡터 #152와 거의 동일한 패턴"
  → tqi#152 시점에 무슨 일이 있었는지 확인
  → 같은 종류의 이상이 반복되는 건지 판단
활용 시나리오 1. 장애 재현 분석: "이 이상 패턴이 과거에도 있었나?" → 유사 벡터 검색
2. 반복 장애 감지: 같은 유사도 패턴이 주기적으로 나타나면 근본 원인 미해결
3. 영향 범위 추적: 같은 시간대 다른 카테고리(.dat2)에서도 이상이 나타나는지 교차 확인
