Ch.1 — 첫 시도: "이걸로 저장 공간을 줄일 수 있지 않을까?"

TurboQuant는 AI 모델의 메모리를 6~8배 줄이는 알고리즘이다. Google이 ICLR 2026에서 발표했고, 학습 없이 바로 적용 가능하다는 점이 매력적이었다.

우리의 가설 WhaTap YARD 서버에 쌓이는 모니터링 데이터(CPU, 메모리, TPS 등)도 숫자 벡터다.
TurboQuant로 압축하면 저장 공간을 대폭 줄일 수 있지 않을까?

실험 설계

YARD의 .dat2 파일에 저장되는 MetricValue(count, sum, min, max, last)를 TurboQuant 4-bit으로 압축한 뒤 복원하여 원본과 비교했다.

결과: 값이 박살났다

메트릭원본TQ 복원차이
CPU (%)52.7%36.8%-15.9
Memory (%)71.3%64.2%-7.1
TPS627564-63
Response Time2007ms1807ms-200ms
문제 CPU가 52.7%인데 대시보드에 36.8%로 표시되면 사용할 수 없다.
모니터링 데이터는 개별 값의 정확도가 생명이다.

이 시점에서 "TurboQuant는 모니터링에 안 맞다"고 결론을 내릴 수도 있었다. 하지만 왜 이렇게 많이 틀어지는지가 궁금했다.

Ch.2 — 왜 안 되나: 회전이 값을 섞어버린다

TurboQuant의 핵심: "빙글빙글 돌리기"

TurboQuant는 데이터를 압축하기 전에 랜덤 회전 행렬로 벡터를 돌린다. 이게 AI에서는 장점이지만, 모니터링에서는 독이 된다.

AI 임베딩 (잘 됨)

모든 차원이 비슷한 스케일 (-1 ~ 1)
[0.12, -0.05, 0.31, 0.08, ...]
회전해도 스케일 유지 → 양자화 정밀

모니터링 데이터 (안 됨)

차원마다 스케일이 완전히 다름
[CPU=73, TPS=500, GC=10]
회전하면 500이 73에 섞임 → 값 파괴

회전이 하는 일

회전 전: [CPU=73, TPS=500, GC=10]

회전 후: y[0] = 0.3×73 + 0.2×500 + 0.5×10 = 126.9
         y[1] = -0.1×73 + 0.6×500 + 0.3×10 = 295.7
         y[2] = 0.8×73 + -0.4×500 + 0.2×10 = -139.6

각 좌표가 모든 원본 값의 선형 조합 → TPS(500)가 모든 곳에 영향
비유: 과일 주스 사과(CPU), 오렌지(TPS), 포도(GC)를 믹서기에 넣고 갈아버린 것과 같다.
양자화(오차)를 거친 후 다시 분리하려 해도, 이미 섞인 주스에서 사과만 정확히 빼낼 수 없다.

스칼라 양자화는 사과를 사과대로, 오렌지를 오렌지대로 따로 보관하는 것이라 원본을 정확히 복원할 수 있다.

이걸 숫자로 확인

28%
TQ 4-bit 오차 (회전O)
0.7%
codebook만 (회전X)
0.5%
스칼라 8-bit

회전을 빼면 오차가 40배 줄어든다. 회전 자체가 문제의 근본 원인이었다.

핵심 발견 TurboQuant의 회전은 "내적(유사도)을 보존"하기 위한 것이지, "개별 값을 보존"하기 위한 것이 아니다. AI에서는 내적이 중요하고, 모니터링에서는 개별 값이 중요하다.

회전이 문제라는 걸 알았지만, 혹시 다른 파라미터를 조정하면 쓸 만해지지 않을까 5가지를 시도했다.

결론 비트를 아무리 올려도, 차원을 바꿔도 한계가 있다.
8-bit(256개 코드북)까지 올려도 7% 오차 — 대시보드에 표시하기엔 부족하다.
회전이 오차를 "교차 전파"시키는 구조적 문제는 코드북 수를 늘린다고 해결되지 않는다.

이 시점에서 선택지는 두 가지였다:

  1. "TurboQuant는 저장에 안 맞으니 포기하자" → 유사도 검색/이상 탐지에만 쓰기
  2. "회전 자체를 바꿀 순 없을까?"

Ch.4 — 전환점: "중심값을 빼고, 범위별로 나눠서 돌리면?"

아이디어 1: 중심값(대표값) 빼기

CPU가 대부분 40~80% 사이라는 걸 안다면, 평균(55%)을 빼고 잔차만 양자화하면 값이 작아져서 오차도 작아지지 않을까?

원본: CPU = 73.12%
↓ 중심값 빼기
잔차: 73.12 - 55 = 18.12 (작은 값!)
↓ 정규화 [0,1]
정규화: 18.12 / 100 = 0.18
↓ TurboQuant 회전 + 양자화
복원: 0.19
↓ 역정규화 + 중심값 더하기
복원: 0.19 × 100 + 55 = 74.0 (오차 0.88!)
28%
TQ 4-bit 그대로
2.3%
중심값 빼기 + TQ 4-bit
0.06%
중심값 + TQ 8-bit

중심값을 빼는 것만으로 12배 개선. 하지만 다차원(CPU+TPS+GC를 같이)에서는 여전히 문제가 있었다.

아이디어 2: 범위별로 나눠서 돌리기

CPU(0~100)와 TPS(0~10000)를 같이 돌리면 섞이니까, 비슷한 범위끼리 그룹으로 묶어서 그룹 안에서만 돌리면 어떨까?

❌ 전체 한 번에 회전

[CPU, Mem, Disk, TPS, RT, GC, Error]
         ↓ 7×7 회전
TPS(500)가 CPU에 섞임 → 파괴

✅ 그룹별 회전

그룹A [CPU, Mem, Disk] → 3×3 회전
  (다 0~100 스케일, 섞여도 OK)
그룹B [TPS, RT] → 2×2 회전
  (다 큰 수끼리만)
그룹C [GC, Error] → 2×2 회전
  (다 작은 수끼리만)
비유: 과일 정리 사과와 오렌지를 믹서기에 같이 넣으면 주스가 돼서 분리 불가능.
사과는 사과끼리, 오렌지는 오렌지끼리 따로 갈면 각각의 맛이 유지된다.

비슷한 크기의 값끼리만 함께 회전하면, 섞여도 큰 문제가 안 된다.

Ch.5 — 결과: 그룹 회전이 답이었다

실측 결과: 7개 메트릭의 상대오차 비교

같은 데이터(10,000샘플)를 4가지 방식으로 압축한 뒤, 복원값이 원본과 얼마나 다른지를 상대오차(%)로 측정했다. 낮을수록 원본에 가깝다.

표 읽는 법 CPU % 행의 "223%"는 원본 대비 복원값이 평균 223% 벗어남을 뜻한다.
예: 원본 50% → 복원값이 50±111.5 범위 → 대시보드에 -61%~161%로 표시될 수 있다.

"0.35%"는 원본 50% → 복원값 49.8~50.2 → 대시보드에서 구분 불가능.
전체 회전 → 그룹+중심+정규화+8-bit
CPU: 223% → 0.35% — 637배 개선
Error: 1659% → 0.36% — 4608배 개선

8-bit에서 모든 메트릭이 0.4% 이하 오차. 대시보드에서 구분 불가능.

더 놀라운 발견: "맞춤 눈금" 방식을 이겼다

비교 대상으로 "적응형 코드북"이라는 방법도 테스트했다. 이건 회전을 아예 하지 않고, 대신 실제 데이터 분포에서 자주 나오는 값들을 대표값(눈금)으로 미리 학습해두는 방식이다.

적응형 코드북이란? 시계 눈금을 균등하게 배치하는 대신, 데이터가 많이 몰리는 구간에 눈금을 촘촘하게 배치하는 것.

예: CPU가 대부분 40~80%라면
  균등 눈금: 0, 6.25, 12.5, 18.75, 25, ... (0~40%에 쓸데없이 많은 눈금)
  적응형 눈금: 14, 31, 42, 49, 54, 58, 62, 66, 70, 75, 81, 88, 96 (40~80%에 집중)

회전 없이 각 메트릭의 값을 가장 가까운 눈금에 매핑하므로, 다른 메트릭 값의 영향을 받지 않는다.

그런데 8-bit에서 "그룹 회전 TurboQuant"가 이 적응형 코드북보다 오히려 더 정확했다.

메트릭그룹+중심 TQ 8-bit적응형 코드북 8-bit (회전X)
TPS0.25%0.50%
GC0.27%0.43%
Error0.36%0.51%
왜 회전이 오히려 도움이 되나? 같은 스케일 그룹 내에서 회전하면, 양자화 오차가 여러 차원에 분산된다.
한 값에 몰리는 대신 여러 값에 조금씩 나눠지므로, 최대 오차가 줄어든다.

회전의 "값 섞임"이 같은 스케일 내에서는 부작용이 아니라 오차 분산 효과로 작용한다.

적응형 코드북(맞춤 눈금) 단독 결과

회전을 쓰지 않고, 메트릭별로 "자주 나오는 값에 눈금을 집중"하는 것만으로도 큰 효과가 있었다. 5-bit(32개 눈금)이면 대부분 메트릭에서 오차 4% 이하.

판단 기준은 두 가지다:
  1. 숫자 배열이 충분히 긴가? (차원 50 이상)
  2. 값들의 크기(스케일)가 비슷한가? (또는 그룹으로 나눌 수 있는가?)

YARD 데이터 적용 가능성

TurboQuant 적용 가능

공통 이유: 원본 값이 아니라 패턴을 비교하는 데이터 아래 데이터들의 공통점은 개별 값의 정확도보다 "전체 모양이 비슷한가?"가 중요하다는 것이다.

히트맵은 120개 버킷의 분포 형태로 "느린 구간이 많은지"를 판단한다. 서버 카운터는 78개 메트릭의 조합 패턴으로 "이 서버가 정상인지"를 판단한다. 퍼센타일은 응답시간의 분포 곡선으로 "이 URL이 느려졌는지"를 판단한다.

이런 분석/통계 데이터는 개별 숫자 하나가 약간 틀려도 전체 패턴은 유지된다. TurboQuant가 정확히 잘하는 것 — "패턴(내적/유사도)을 보존하면서 압축" — 이 여기에 딱 맞는다.

1. 히트맵 (HitMapPack) — 최적 대상

120
차원 (응답시간 버킷)
~7x
예상 압축률 (4-bit)
248B
히트맵 1건 압축 크기

왜 적합한가: 120개 값이 모두 "건수(count)"로 같은 스케일. 회전해도 섞이는 문제가 없다.

가능한 기능:

  • "지금 히트맵과 비슷한 모양의 과거 시점 찾기" — 장애 재현 분석에 활용
  • "히트맵 분포가 갑자기 바뀌면 알림" — 응답시간 패턴 이상 탐지
  • "A 서비스의 히트맵과 비슷한 패턴의 다른 서비스" — 장애 영향 범위 추적
비즈니스 임팩트 현재 히트맵은 "눈으로 보고 비교"해야 한다. 이 기능이 있으면 "지난주 화요일 오후 3시에도 이런 패턴이었습니다"를 자동으로 알려줄 수 있다. 장애 분석 시간을 크게 줄일 수 있는 기능이다.

2. 서버 카운터 (CounterPack) — 그룹 회전 적용

78
차원 (메트릭 필드 수)
0.35%
그룹 회전 8-bit 오차
100%
PoC 이상 탐지 재현율

왜 적합한가: CPU/메모리/TPS 등 스케일이 다르지만, 비슷한 범위끼리 그룹으로 나눠 회전하면 해결된다.

가능한 기능:

  • "이 서버와 메트릭 패턴이 비슷한 서버 목록" — 같은 역할의 서버 자동 그룹핑
  • "CPU는 정상인데 TPS와 GC가 동시에 이상" 같은 복합 이상 탐지 — 단일 threshold 알림으로는 못 잡는 것
  • 장애 발생 시 "영향 받은 서버 자동 식별" — 비슷한 패턴 변화를 보이는 서버 즉시 리스트업
비즈니스 임팩트 현재 WhaTap 알림은 "CPU > 90%"처럼 단일 메트릭 threshold다. 하지만 실제 장애는 "CPU는 70%인데 GC가 치솟고 TPS가 떨어지는" 복합 패턴인 경우가 많다. 78개 메트릭을 동시에 보는 벡터 기반 탐지는 기존 알림이 놓치는 장애를 잡을 수 있다.

50~100
차원 (정렬된 float)
~6x
예상 압축률

왜 적합한가: 모두 "밀리초(ms)" 단위로 같은 스케일. 정렬된 float 배열이라 분포의 "모양"을 벡터로 표현할 수 있다.

가능한 기능:

  • "이 URL과 응답시간 분포가 비슷한 URL" — 느린 API 패턴 분석
  • "P99가 갑자기 바뀐 URL 자동 감지" — 퍼센타일 분포 형태 변화 탐지
비즈니스 임팩트 특정 URL이 느려졌을 때 "이 URL과 비슷한 패턴의 다른 URL도 느려졌나?"를 즉시 확인할 수 있다. 동일 DB를 사용하는 API들이 함께 느려지는 패턴을 자동으로 묶어서 근본 원인을 빠르게 좁힐 수 있다.

TurboQuant 적용 불가

공통 이유: 원본 값을 그대로 돌려줘야 한다 아래 데이터들의 공통점은 API가 저장된 값을 원본 그대로 사용자에게 보여준다는 것이다.

대시보드에 "CPU 52.7%"라고 표시해야 하는데 "36.8%"가 나오면 안 된다. SQL 문이 SELECT * FROM users인데 글자가 바뀌면 안 된다. 인덱스 위치가 1바이트라도 틀리면 데이터를 아예 못 찾는다.

TurboQuant는 "패턴이 비슷한가?"를 비교하는 용도로 만들어졌지, "원본을 정확히 복원하는 용도"가 아니다. 원본 저장이 목적인 데이터에는 구조적으로 맞지 않는다.
데이터왜 안 되나대안
차원별 메트릭 (TagCountPack .dat2)숫자 5개뿐(저차원). API가 개별 값을 직접 표시.맞춤 눈금 5~8-bit
통계 집계 (StatGeneralPack)행 수가 가변적이고, 컬럼마다 스케일이 다름.컬럼별 맞춤 눈금
호출 스택 (ActiveStackPack)해시값 시퀀스. 양자화하면 다른 함수를 가리킴.별도 임베딩 필요
트랜잭션 프로필 (ProfilePack)바이너리 스텝 시퀀스. 1비트 바뀌면 다른 실행 과정.별도 임베딩 필요
텍스트 (hvtxt)URL, SQL 등 문자열. 글자가 바뀌면 의미가 달라짐.LZ4, Zstd 압축
구조화 로그 (LogSinkPack)숫자 필드 3~8개(저차원). 본문은 문자열.숫자: 맞춤 눈금 / 문자열: Zstd
인덱스/메타 (.tim, .tags, .dat)위치 포인터. 1비트 틀리면 데이터를 못 찾음.변경 불가

이 PoC에서 배운 것

교훈 1. 알고리즘의 설계 목적을 이해해야 한다.
TurboQuant는 "내적 보존"을 위해 만들어졌다. 이걸 "값 보존"에 쓰려 하면 안 된다.

2. "안 된다"에서 멈추지 않으면 길이 보인다.
전체 회전 → 그룹 회전으로 바꾸니 오차가 637배 줄었다. 구조를 이해하면 해결책이 나온다.

3. 데이터 특성에 맞게 변형해야 한다.
AI 임베딩에 최적화된 알고리즘을 모니터링 데이터에 그대로 적용하면 안 된다. 중심값 빼기, 범위별 그룹핑, 적응형 코드북 같은 도메인 맞춤 변형이 필요하다.

Ch.8 — 벡터 인덱스를 파일로 저장하기

Ch.7까지는 벡터 인덱스가 메모리에만 있었다. YARD가 재시작되면 다 날아간다. 실서비스에서 쓰려면 파일로 저장하고, 재시작 시 복원할 수 있어야 한다.

.tqi 파일 포맷

YARD의 yardbase 파일 구조와 같은 패턴으로, 벡터 인덱스 전용 파일 포맷(.tqi)을 만들었다.

yardbase/1/20260330/02/
├── tagc/           ← 기존 메트릭
├── db3tx/          ← 기존 히트맵
└── tqindex/        ← 벡터 인덱스 (신규)
    └── hitmap.tqi

.tqi 파일 내부:
┌──── 헤더 (20바이트) ────┐
│ magic: "TQI1"           │
│ dim: 120, bits: 4       │
│ compressedSize: 64      │
│ rotationSeed: 42        │
├──── 엔트리 반복 ────────┤
│ time (8B)               │
│ pcode (8B)              │
│ oid (4B)                │
│ totalHits (4B)          │
│ compressed (64B)        │
│ ── 엔트리당 88바이트 ── │
└─────────────────────────┘
설계 원칙 Append-only: 새 데이터는 파일 끝에 추가만 한다. 기존 데이터를 수정하지 않으므로 안전하다.
고정 크기 엔트리: 모든 엔트리가 88바이트로 동일. n번째 엔트리 위치 = 20 + n × 88. 랜덤 액세스가 가능하다.
재시작 복원: 파일을 열고 헤더만 읽으면 끝. 엔트리 수 = (파일 크기 - 20) / 88. 0.16ms면 완료.
