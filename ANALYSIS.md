## 1. TurboQuant란 무엇인가

Google Research가 ICLR 2026에서 발표한 **벡터 양자화(Vector Quantization)** 알고리즘입니다.
LLM의 Key-Value Cache를 3~4비트로 압축하여 GPU 메모리를 6~8배 절약하는 것이 원래 목적입니다.

### 알고리즘 핵심 (2단계)

| 단계 | 연산 | 목적 |
|------|------|------|
| **1단계** | 랜덤 직교 회전(QR 분해) + Lloyd-Max 최적 양자화 | 좌표를 Beta 분포로 변환 후 MSE-최적 코드북으로 양자화 |
| **2단계** | QJL 1-bit 잔차 프로젝션 | 양자화 잔차의 부호만 저장하여 내적 편향 보정 |

### TurboQuant가 잘하는 것

**고차원 벡터(수백~수천 차원)의 내적을 보존**하면서 극단적으로 압축합니다.
개별 값이 약간 틀려도, 벡터 간 유사도(내적/코사인)는 정확하게 유지됩니다.

### TurboQuant가 못하는 것

**개별 값의 정확한 복원**은 보장하지 않습니다.
랜덤 회전 행렬이 값을 "섞기" 때문에 입력 52.7이 복원 시 36.8이 될 수 있습니다.
차원이 낮을수록(5~20) 이 문제가 심각해집니다.

### AI에서는 왜 스칼라 8-bit 대신 TurboQuant를 쓰는가?

4096차원 벡터에서 스칼라 양자화의 작은 오차(±0.2%)가 내적 계산 시 누적됩니다:

```
내적 오차 ≈ 개별오차 × √차원 = 0.2% × √4096 ≈ 64배 증폭
```

TurboQuant는 **회전으로 오차를 균등 분배**하고 **QJL로 내적 편향을 보정**하여 이 누적 문제를 해결합니다.

모니터링 데이터는 내적이 아니라 개별 값을 읽으므로, 이 장점이 필요하지 않습니다.

## 2. WhaTap YARD 데이터 구조

### YARD란?

WhaTap 모니터링 플랫폼의 핵심 데이터 수집/저장 서버입니다.
에이전트 → TCP:6610 → yardbase(시계열 DB) → API:7710 → 대시보드.

### yardbase 디렉토리 구조

```
yardbase/
├── {pcode}/                          ← 프로젝트별
│   ├── {YYYYMMDD}/                   ← 일별
│   │   ├── {HH}/                     ← 시간별
│   │   │   ├── tagc/                 ← 차원별 메트릭 (TagCountPack)
│   │   │   │   ├── *.dat             ← 메인 팩 (헤더+인덱스)
│   │   │   │   ├── *.dat2            ← field 데이터 (MetricValue) ★ 압축 대상
│   │   │   │   ├── *.dat3            ← tag 데이터 (문자열)
│   │   │   │   ├── *.tim             ← 시간 인덱스
│   │   │   │   ├── *.tags            ← 태그 메타데이터
│   │   │   │   └── *.tdata           ← 시계열 데이터
│   │   │   ├── db3/                  ← 통계 집계 (StatGeneralPack)
│   │   │   ├── db3tx/                ← 히트맵 + 프로필 (HitMapPack)
│   │   │   └── hvtxt/                ← 텍스트 해시 (URL, SQL)
│   │   └── meta-inf/
│   └── 20000101/                     ← config, kvdb, longTerm
└── logsink1/                         ← 구조화 로그
```

### 주요 데이터 팩 유형

| 팩 | 저장 위치 | 내용 | 차원 | 데이터 형태 |
|---|---------|------|------|-----------|
| **HitMapPack1** | db3tx/ | 응답시간 분포 히스토그램 | **120** | int[120] (동일 스케일) |
| **HitMapRumPack** | db3tx/ | RUM 페이지 로드 분포 | **120** | int[120] (동일 스케일) |
| **CounterPack** | 캐시/큐브 | 서버 상태 (CPU,메모리,TPS...) | **78** | float/int 혼합 |
| **TagCountPack** | tagc/*.dat2 | 차원별 메트릭 (MetricValue) | 5 | count,sum,min,max,last |
| **StatGeneralPack** | db3/ | 통계 집계 테이블 | 가변 | AnyList 컬럼 |
| **KLL Quantile** | 큐브 | 응답시간 퍼센타일 스케치 | 50~100 | sorted float[] |
| **ActiveStackPack** | 캐시 | 활성 호출 스택 | 가변 | int[] 해시 시퀀스 |
| **ProfilePack** | db3tx/ | 트랜잭션 프로필 스텝 | 가변 | byte[] 바이너리 |
| **LogSinkPack** | logsink1/ | 구조화 로그 | 3~8 | tags(문자열) + fields(숫자) |

## 3. YARD 데이터 분류: 적용 가능 vs 불가능

### 적용 가능

| 데이터 | 차원 | 스케일 | 적용 방식 | 기대 효과 |
|--------|------|--------|----------|----------|
| **히트맵 (HitMapPack1)** | 120 | 동일(count) | 분포 벡터 직접 양자화 | 히트맵 패턴 비교, 이상 시점 탐지 |
| **RUM 히트맵 (HitMapRumPack)** | 120 | 동일(count) | 동일 | 페이지 로드 분포 비교 |
| **CounterPack (서버 상태)** | 78 | 혼합 (정규화 필요) | z-score 정규화 후 벡터화 | 서버 패턴 유사도 검색, 이상 탐지 |
| **KLL Quantile** | 50~100 | 동일(ms) | 퍼센타일 분포 벡터화 | "이 URL과 응답 분포가 비슷한 URL" 검색 |
| **StatGeneralPack** | 가변 | 혼합 | 행 단위 정규화+벡터화 | 유사 URL/SQL 자동 그룹핑 |
| **ActiveStackPack** | 가변 | 동일(hash) | 해시 시퀀스 임베딩 | 유사 호출 스택 클러스터링 |

### 적용 불가능

| 데이터 | 차원 | 불가 사유 | 대안 |
|--------|------|----------|------|
| **TagCount .dat2 (MetricValue)** | 5 | 저차원 + 이종 스케일, 회전이 값 섞어 오차 20%+ | 스칼라 8-bit |
| **TagCount .dat3 (태그 텍스트)** | - | 문자열, 숫자 아님 | 해시/사전 압축 |
| **hvtxt (URL/SQL 텍스트)** | - | 해시→텍스트 매핑 | LZ4/Zstd |
| **타임스탬프 (Decimal)** | 1 | epoch ms 단일값 | delta encoding |
| **ProfilePack (byte[] steps)** | 가변 | 바이너리 시퀀스, 직접 양자화 불가 | Step 임베딩 선행 필요 |
| **.tim / .tags / .dat (인덱스)** | - | 위치 포인터/메타, 정확해야 함 | 변경 불가 |
| **logsink (로그 본문)** | - | 문자열 | Zstd |
| **logsink fields (숫자)** | 3~8 | 저차원 | 스칼라 8-bit |
| **정수 전용 (count=1)** | 1 | 이미 정확, 양자화 무의미 | 유지 |
| **config / kvdb** | - | 설정 데이터 | 변경 불가 |

### 판단 기준

**TurboQuant에 맞는 데이터:**
- 차원 50 이상
- 동일하거나 비슷한 스케일
- 개별 값보다 **패턴/분포**가 중요
- 유사도 비교(내적)에 사용

**TurboQuant에 안 맞는 데이터:**
- 차원 20 이하
- 스케일이 크게 다른 필드 혼합
- 개별 값 정확도가 중요 (API 응답에 직접 표시)
- 문자열 / 바이너리 / 인덱스

## 5. 핵심 발견: 왜 저장 압축에는 안 맞는가

압축률 8.57x를 달성했지만, API 응답값이 크게 달라집니다:

| 메트릭 | 원본 | TQ 복원 | 차이 |
|--------|------|--------|------|
| CPU (%) | 52.7% | **36.8%** | -15.9 |
| Memory (%) | 71.3% | **64.2%** | -7.1 |
| TPS | 627 | **564** | -63 |
| Response Time | 2007ms | **1807ms** | -200 |

비교 — 스칼라 8-bit:

| 메트릭 | 원본 | 8-bit 복원 | 차이 |
|--------|------|-----------|------|
| CPU (%) | 52.7% | 52.5% | -0.2 |
| Memory (%) | 71.3% | 71.4% | +0.1 |
| TPS | 627 | 625 | -2 |

**원인:** 랜덤 회전 행렬이 서로 다른 스케일(CPU 0~100, TPS 0~10000)의 값을 섞습니다.
LLM 임베딩은 모든 차원이 비슷한 스케일(-1~1)이라 이 문제가 없습니다.

## 6. 튜닝 시도와 한계

5가지 튜닝을 시도했습니다:

| 튜닝 방법 | 상대오차 | 압축률 | 결과 |
|----------|---------|--------|------|
| 현재: TQ 4-bit, dim=5 | 28.17% | 2.9x | 부족 |
| 비트 증가: TQ 6-bit | 10.72% | 2.5x | 여전히 부족 |
| 비트 증가: TQ 8-bit | 7.13% | 2.2x | 여전히 부족 |
| 시간 배치: dim=100 | 39.50% | 7.4x | 더 나빠짐 |
| 델타 인코딩 + TQ | 2200% | 6.1x | 완전 파탄 |
| **회전 제거: codebook만** | **0.69%** | 1.2x | 정확 (압축 낮음) |
| **스칼라 8-bit** | **0.49%** | **4.0x** | **최적** |

**결론:** 회전 행렬이 오차의 근본 원인. 회전을 빼면 TurboQuant가 아니라 그냥 codebook 양자화.
모니터링 데이터 저장에는 단순한 스칼라 양자화가 구조적으로 최적.

## 7. 실제 WhaTap 데이터 분석

로컬 yardbase의 .dat2 파일 149개(639 레코드, 47 카테고리)를 파싱 분석:

### 데이터 특성

| 특성 | 상세 | 영향 |
|------|------|------|
| 타입 다양성 | MetricValue, Decimal, Double, Float, BlobValue 혼재 | 일괄 양자화 불가 |
| 타임스탬프 혼입 | Decimal 필드에 epoch ms 포함 | 양자화 대상에서 제외 필요 |
| 좁은 범위 | count=1~2 같은 상수 필드 다수 | 양자화 이점 없음 |
| 희소 분포 | 대부분 0, 간헐적 큰 값 | 균등 양자화보다 log-scale 적합 |

### 실데이터 양자화 오차

| 카테고리.필드 | 값 범위 | 스칼라 8-bit | TQ 8-bit |
|-------------|---------|------------|---------|
| page_load.로딩시간 | 27~317ms | **0.22%** | 8.74% |
| resource.전송량 | 0~5350 | **6.06%** | 184.22% |
| web_vitals.LCP | 0~80ms | **0.13%** | 10.94% |
| screen_loading | 가변 | **1.06%** | 27.41% |

모든 필드에서 스칼라 8-bit이 TurboQuant보다 10~300배 정확합니다.

## 8. 최종 적용 전략

### 역할 분담: 저장은 스칼라, 분석은 TurboQuant

```
[저장 경로]
에이전트 데이터 → 스칼라 8-bit → .dat2 저장 (4x 압축, 오차 0.2%)
                                    ↓
                              API 조회 (복원: 곱하기 1번)

[분석 경로 — 디스크에 별도 저장 불필요]
.dat2 → 8-bit 복원 → 최근 5분 메트릭 수집
                         ↓
                    480차원 벡터 조립 (8메트릭 × 60타임스텝)
                         ↓
                    TurboQuant 압축 → 메모리 인덱스 (248B/벡터)
                         ↓
                    베이스라인 비교 → 이상 시 EventPack 발생
```

### 1순위: 히트맵 (HitMapPack)

120차원, 동일 스케일(count), 패턴 비교 목적 — TurboQuant가 설계된 정확한 시나리오.

가능한 기능:
- "지금 히트맵과 비슷한 과거 시점 찾기" (장애 재현 분석)
- "히트맵 분포가 갑자기 바뀌면 알림" (응답시간 분포 이상 탐지)
- "A 서비스의 히트맵과 비슷한 패턴의 다른 서비스" (영향 범위 추적)

### 성능 영향

| | 스칼라 8-bit (저장) | TurboQuant (분석) |
|--|-------------------|------------------|
| 복원 속도 | ~0 ns | 289 ns/record |
| 1000 레코드 조회 | ~0 ms | 0.29 ms |
| API 응답 영향 | 없음 | 0.29% |
| 저장 공간 | 4x 압축 | 메모리 only |
